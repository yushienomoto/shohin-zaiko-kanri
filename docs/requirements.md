## 1. 背景・目的

### 1.1 背景

- 倉庫消耗品の発注は、現状「なんとなく少ない」という感覚に依存する部分があり、安全在庫、発注点、発注数量が十分に共有・記録されていない。
- 原油・原材料の輸入停止や供給制約等が発生した場合、施工・倉庫業務に必要な消耗品が欠品し、業務継続へ影響する可能性がある。
- 発注担当は経験の浅い社員を想定しており、担当者個人の判断を極力減らし、定めたルールに沿って処理できる仕組みが必要である。
- 一般社員による早期警報と、担当者による定期的な数量確認を分け、確認負荷と欠品リスクを両立して抑える必要がある。

### 1.2 目的

- 一般社員が、置き場のQRコードから不足品を簡単に報告できるようにする。
- 重要度に応じた確認頻度を自動化し、担当者の毎週の確認作業を必要範囲に限定する。
- 在庫数と発注点を比較し、標準発注数量を自動採用することで、発注判断の属人化と入力ミスを減らす。
- 仕入先別の社内発注依頼文を自動生成し、ワンクリックで全文コピーできるようにする。
- 不足報告、在庫確認、発注依頼、商品設定変更の履歴を半永久的に保存し、発注点・重要度・標準発注数の見直しに活用する。

## 2. 対象範囲と前提

### 2.1 初期対象

| 区分 | 内容 |
|---|---|
| 商品 | 商品マスター記載の32品目 |
| 置き場 | A・B・Cの3か所 |
| 発注先 | 大矢荘平商店、ASKUL、ホームセンター、その他マスター登録先 |
| 端末 | 一般社員：iPhone中心／担当者・管理者：Windows PC中心 |
| 保存先 | Googleスプレッドシートを簡易DBとして使用 |

### 2.2 技術方針

- 画面はHTML・CSS・JavaScriptで構成し、GitHub Pages等の静的ホスティングへ配置する。
- データの登録・取得・認証判定はGoogle Apps Scriptを経由し、Googleスプレッドシートへ保存する。
- Python／Flask／Renderは初期版では使用せず、無料プランのスリープ起動待ちを回避する。
- 一般社員用、担当者用、管理者用の画面を同一Webサイト内に持つ。

## 3. 利用者・権限

| 利用者 | 認証 | 主な操作 |
|---|---|---|
| 一般社員 | 認証なし（報告者名は入力） | A・B・Cの商品閲覧、不足報告、報告済み確認、少ない→在庫なしへの更新 |
| 発注担当者 | 担当者用簡易PIN | 確認対象の表示、数量入力・修正、発注候補確認、文面生成・コピー、発注依頼済み処理 |
| 管理者 | 管理者用簡易PIN | 商品マスター編集、履歴検索、設定変更、履歴訂正、CSV出力、分析 |

- 管理者は1名を想定する。
- PINはHTML／JavaScriptへ平文で埋め込まず、Google Apps Script側で判定する。
- 認証状態は同一ブラウザ内で一定時間保持し、一定時間後は再認証する。

## 4. 全体業務フロー

1. 一般社員がQRコードまたはホーム画面から場所A・B・Cを開く。
2. 発注点より少ないと感じた商品を選び、「少ない」または「在庫なし」で報告する。
3. システムは、重要度A、当月未確認の重要度B、不足報告品を統合して担当者の確認リストを作成する。
4. 担当者が現物数量を入力し、発注点以下なら標準発注数で発注候補へ自動追加する。
5. すべての確認後、担当者が「発注する」を押し、発注先別の社内依頼文またはホームセンター記入リストを生成する。
6. メール対象は全文コピー後、担当者がメールを送信し、「メールを送信しました」にチェックして「発注依頼済み」にする。
7. ホームセンター対象は倉庫内ホワイトボードへ記入後、「ホワイトボードへ記入しました」にチェックして「記入済み」にする。
8. 各操作と当時の設定値を履歴へ保存する。

## 5. 機能要件

### 5.1 一般社員向け不足報告

- ホーム画面から場所A・B・Cへ遷移できること。
- 各場所のQRコードから、該当場所の画面を直接開けること。画面上のメニューから他の場所へ移動できること。
- 場所Aは「テープ・固定用品」「梱包・施工用品」「清掃・共用品」の分類見出しで表示すること。場所B・Cは品目一覧表示とすること。
- 商品名、発注点、単位を表示すること。
- 一度に複数商品を選択し、各商品を「少ない」または「在庫なし」でまとめて送信できること。
- 報告者名を入力できること。前回入力名をブラウザへ保存し、次回は自動表示してよい。
- 未処理の報告がある商品は「報告済み」と表示し、報告日時・状態を確認できること。
- 報告済みの商品は重複報告を抑止すること。ただし「少ない」から「在庫なし」への更新は許可すること。

### 5.2 担当者向け在庫確認

- 担当者名を入力できること。
- 「今週の確認」「発注候補」「全商品」「履歴」の主要メニューを用意すること。
- 今週の確認対象は、(1) 在庫なし報告、(2) 少ない報告、(3) 重要度A定期確認、(4) 重要度B月次確認の順で優先表示すること。
- 同一商品が複数条件に該当しても1件に統合し、該当理由を併記すること。
- 商品ごとに場所、重要度、発注点、単位、前回確認数、前回確認日、報告情報を表示すること。
- 現在数量は商品マスターの単位で入力すること。
- 数量入力後も修正可能とし、修正日時・修正前後・操作者を履歴へ残すこと。
- 修正後の数量に応じて、発注候補への追加・除外を自動再判定すること。
- 未確認品が残る場合、発注文面作成前に警告を表示すること。

### 5.3 発注候補判定

- 現在数量が発注点以下の場合、商品を発注候補へ自動追加すること。
- 発注数量は商品マスターの標準発注数を自動採用し、担当者画面では原則変更不可とすること。
- 標準数量と異なる発注が必要な場合は、管理者または上司へ相談し、初期版では生成後のメール本文で変更する運用とすること。
- 実際の発注数量が標準と異なる場合、発注完了登録時に実発注数と変更理由等を記録可能とすること。

### 5.4 発注依頼文・処理完了

- 全確認後、担当者がPCで「発注する」を押すと、発注先ごとに内容を生成すること。
- 商品名は「正式名称」が登録されている場合は正式名称を優先し、空欄の場合は「品名」を使用すること。
- 発注依頼文はプレーンテキストで生成し、仕入先ごとに「全文コピー」ボタンを設けること。
- コピー成功後は「コピーしました」と明示すること。
- メール対象は文面生成時に「文面作成済み」と記録し、コピーのみでは「発注依頼済み」にしないこと。
- メール送信後、「メールを送信しました」チェックを必須とし、確認ダイアログ後に仕入先単位で「発注依頼済み」にすること。
- ホームセンター品はメールを生成せず、ホワイトボード記入リストを表示すること。記入後のチェックと確認操作を完了トリガーとすること。
- 仕入先ごとに個別に完了でき、未完了の処理は担当者ホームへ残すこと。

### 5.5 管理者機能

- 商品マスターの追加、編集、使用停止（廃止）ができること。
- 場所、分類、重要度、発注点、標準発注数、単位、発注先、正式名称、URL、備考等を編集できること。
- 設定変更前後を確認画面に表示し、変更者・変更日時・変更理由を履歴へ保存すること。
- 不足報告、在庫確認、文面作成、発注依頼、ホワイトボード記入、マスター変更の各履歴を検索できること。
- 期間、年、商品、重要度、仕入先、担当者、状態等で絞り込みできること。
- 検索結果をCSV出力できること。
- 初期版では高度な分析画面を必須とせず、データ蓄積後に平均発注間隔、発注回数、見直し候補等を追加できる構造とすること。

## 6. 業務ルール

| 重要度 | 定期確認 | 不足報告がある場合 |
|---|---|---|
| A | 毎週、数量確認 | 報告表示を強調し、必ず確認 |
| B | 月1回、数量確認 | 定期日を待たず必ず確認 |
| C | 定期確認なし | 必ず確認 |

- 重要度Bは、月が変わると当月未確認状態とする。月初の固定日ではなく、その月最初の在庫確認時に確認対象へ追加し、完了するまで翌週以降も表示する。
- 長期休暇、担当者不在、繁忙等で確認できなかった場合も、Bの未確認状態を翌回へ持ち越す。
- 一般社員の不足報告は発注確定ではなく、担当者への早期警報・確認依頼として扱う。
- 発注点・標準発注数・重要度の変更権限は管理者に限定する。
- 発注依頼済みと外部仕入先への実発注完了は同義ではない。本システムでは社内担当者へ依頼した時点を記録する。

## 7. データ要件

### 7.1 データ保存方針

- Googleスプレッドシートを簡易DBとして使用し、年次で物理分割せず各履歴シートへ追記する。
- 履歴は原則削除せず半永久的に保存する。必要に応じ、将来は古いデータを別ファイルへアーカイブできる。
- 表示時は全件取得を避け、最新件数または検索条件に合う範囲のみ取得する。
- 商品マスターを変更しても過去の履歴が変わらないよう、発生時点の発注点、重要度、標準発注数、単位等を履歴側へ複製保存する。

### 7.2 想定シート

| シート | 主な内容 |
|---|---|
| 商品マスター | 場所、品名、発注先、重要度、発注点、発注数、単位、備考、値段、正式名称、URL、使用状態等 |
| 不足報告履歴 | 報告日時、商品、状態、報告者、処理状態、更新履歴 |
| 在庫確認履歴 | 確認日時、商品、現在数、当時の発注点、判定、確認者、修正履歴 |
| 発注処理履歴 | 文面作成日時、コピー日時、発注依頼済み日時、仕入先、担当者、状態 |
| 発注明細履歴 | 商品、確認数、発注点、標準発注数、実発注数、当時の重要度、仕入先 |
| マスター変更履歴 | 変更日時、商品、項目、変更前、変更後、変更者、理由 |

### 7.3 容量と検索

- 年間約1,600件以上の確認履歴を想定しても、Googleスプレッドシートのセル上限に対して十分余裕がある。
- 年別・期間別表示は、日付列を条件に管理画面側で絞り込む。シートを年ごとに分ける必要はない。
- 期間は暦年、任意期間、過去12か月等の検索に対応できる構造とする。

## 8. 画面要件

| 画面 | 主要要素 |
|---|---|
| ホーム | A・B・Cへの導線、一般報告、担当者ログイン、管理者ログイン |
| 場所A/B/C | 分類見出し、商品名、発注点、単位、報告済み状態、複数選択、報告者名、送信 |
| 担当者ホーム | 未確認件数、確認済件数、発注候補件数、文面作成済み・未完了の警告 |
| 今週の確認 | 優先順位付き確認リスト、数量入力、修正、確認完了 |
| 発注候補 | 仕入先別集約、標準発注数量、確認数、発注点 |
| 発注処理 | 仕入先別テキスト、全文コピー、送信確認、完了ボタン／ホームセンター記入確認 |
| 管理者 | 商品マスター、各履歴、検索、CSV出力、設定変更 |

- iPhoneでは片手操作を想定し、タップ領域を大きく、文字とボタンを明瞭にする。
- Windows PCでは一覧性を優先し、仕入先別カード、検索、コピー操作を分かりやすく配置する。
- 完了・コピー・警告・エラーは色だけに依存せず、文字とアイコンでも示す。

## 9. 非機能要件

| 分類 | 要件 |
|---|---|
| 性能 | QR読取後の初期画面は静的配信により速やかに表示する。全履歴を毎回取得しない。 |
| 可用性 | Render無料枠のスリープに依存しない。Googleサービス障害時は登録できない旨を表示する。 |
| 操作性 | 一般社員は最小操作、担当者は判断不要、管理者は検索・変更可能という役割別設計とする。 |
| 安全性 | 担当者・管理者は別PIN。PIN判定はApps Script側。重要操作は確認ダイアログを挟む。 |
| 監査性 | 数量修正、設定変更、発注処理の前後を履歴へ残す。 |
| 保守性 | 商品や設定はコードへ直書きせず、商品マスターから読み込む。 |
| 互換性 | iPhone SafariおよびWindowsの主要ブラウザで利用できること。 |

## 10. 運用・保守要件

- 初期運用開始後、入力負荷、確認漏れ、誤発注、画面の分かりにくさを確認し、仕様を随時見直す。
- 商品マスターの発注点・重要度・標準発注数は、履歴データを根拠に管理者が見直す。
- 重要度Aでも発注頻度が低い商品はBへ、発注頻度が高い商品は発注点・標準発注数の引き上げを検討する。
- Googleスプレッドシートは定期的にバックアップまたはコピーを取得することを推奨する。
- QRコードは場所A・B・CのURLを示し、商品入替があってもQRを再発行せずマスター変更で対応できるようにする。

## 11. 初期版の対象外・将来拡張

- Gmail／Outlook APIによるメール自動送信および送信済みメールの自動検知
- 外部仕入先への発注完了・納品完了・入庫数量の自動追跡
- バーコード・画像認識による自動在庫カウント
- 高度な需要予測、安全在庫の自動算出、発注点の自動提案
- Googleアカウント単位の本格的な権限管理
- リアルタイム通知、プッシュ通知、メール督促
- 管理ダッシュボードのグラフ・KPI表示

## 12. 受入条件

- 場所A・B・CのQRから該当画面が開き、ホーム経由で相互移動できる。
- 一般社員が複数商品を「少ない／在庫なし」で報告でき、未処理商品に報告済み表示が出る。
- 重要度A、当月未確認B、不足報告品が正しく担当者の確認リストへ統合される。
- 数量入力・修正に応じて、発注候補が自動追加・除外される。
- 標準発注数は担当者が通常操作で変更できない。
- 仕入先ごとの本文が指定テンプレートで生成され、全文コピーできる。
- ホームセンター品はホワイトボード記入確認で処理済みにできる。
- 文面作成済みと発注依頼済みが分離され、未完了処理が残る。
- 各履歴が保存され、管理者画面で年・期間・商品等の条件で検索できる。
- 管理者が商品マスターを変更でき、変更履歴が残る。

## 付録A. 初期商品マスター概要

| 項目 | 件数・内容 |
|---|---|
| 総品目数 | 32 |
| 場所別 | A：25品／B：5品／C：2品 |
| 重要度別 | A：13品／B：9品／C：10品 |
| 発注先別 | ASKUL：16品／ダイソー：1品／ニップ：1品／ホームセンター：2品／大矢荘平商店：11品／楽天：1品 |
| マスター列 | 場所、品名、発注先、重要度、発注点、発注数、単位、備考、値段、正式名称、URL |

### A.1 マスター項目の利用ルール

- 正式名称が入力済みの場合、発注依頼文には正式名称を使用する。
- 正式名称が空欄の場合、品名を使用する。
- URLはASKUL等、商品特定に必要な発注依頼文へ表示する。URLが空欄の場合はURL行を出力しない。
- 単位は数量入力、発注点表示、発注依頼文へ一貫して使用する。

```
・{商品名}　{発注数量}{単位}

```

```
ホームセンター購入品
```
